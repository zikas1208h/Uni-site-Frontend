3. Authentication

All Users
🔓

Login

  • User enters email + password on the Login page
  • Frontend calls POST /api/auth/login with credentials
  • Backend looks up user by email, compares bcrypt-hashed password
  • On success: generates a signed JWT with userId and role payload, returns it to the frontend
  • Frontend stores the token in localStorage and injects it into every subsequent request via Axios interceptor
  • User is redirected to their role-specific dashboard automatically
// Auth flow (simplified)
POST /api/auth/login → bcrypt.compare(pw, hash)
→ jwt.sign({ userId, role }, SECRET, { expiresIn: '7d' })
→ localStorage.setItem('token', jwt)
→ axios.interceptors: Authorization: Bearer <token>
🔒

Password Management

  • Change Password (Student/Staff): POST /api/students/profile/change-password — requires current password verification before hashing and saving the new one
  • Admin Reset: POST /api/students/:id/reset-password — admin can set any student's password directly (no current password required)
  • All passwords are hashed with bcrypt (salt rounds: 10) before storage — never stored in plain text

4. Student Features

Student
🏠

Student Dashboard

The dashboard is the first page a student sees after login. It aggregates data from multiple APIs in a single view.

  • Shows student name, student ID, academic year, semester, lecture group and section number
  • Displays current GPA calculated from all graded courses
  • Shows enrolled course count, completed assignments count, and pending assignments
  • Quick links to Schedule, Courses, Grades, and Assignments
  • Recent notifications panel (assignment posted, grade released)
📚

Course Registration & Enrollment

Students can browse and register for courses during active registration periods only.

  • Call GET /api/courses to list all available courses filtered by the student's academic year
  • Admin opens a Registration Period (POST /api/registration) with start/end date and an allowed course list
  • Frontend checks GET /api/registration/active — if no active period, the Enroll button is disabled
  • Student clicks Enroll → POST /api/courses/:id/enroll → backend validates credit hour limits, period validity, and adds the course to User.enrolledCourses[]
  • Credit hour eligibility check: GET /api/courses/eligibility/credit-hours ensures the student does not exceed the maximum allowed credit hours per semester
📊

Grades & GPA

  • GET /api/grades/student — returns all grades for the logged-in student
  • GET /api/grades/gpa — calculates weighted GPA from all graded courses and returns it
  • Each grade record includes: course code, course name, marks, grade letter (A+/A/B+…), and semester
  • GPA is calculated as: Σ(grade_points × credit_hours) / Σ(credit_hours)
  • Grade distribution chart shows visual breakdown of grade letters across all courses
// GPA calculation logic
gradePoints = { A+: 4.0, A: 3.7, B+: 3.3, B: 3.0, ... }
GPA = Σ(points × creditHours) / Σ(creditHours)
📅

Personal Schedule

Each student's schedule is a filtered view of the master schedule based on their lecture group and section.

  • Student calls POST /api/schedule/generate/me or admin force-pushes it
  • Backend reads the master schedule, filters slots where slot.group === student.lectureGroup for lectures and slot.group === student.lectureGroup AND slot.section === student.section for sections
  • Result is saved as a Schedule document linked to the student
  • View shows week-grid and list views, togglable
  • Each slot shows: course name/code, type (Lecture/Section), Group number, Section number, venue, staff name
  • Days shown: Sunday through Thursday + Saturday (6 working days)
📝

Assignments & Submissions

  • GET /api/assignments/my — returns all assignments for courses the student is enrolled in
  • Each assignment shows: title, course, due date, description, attached file (if any)
  • Student submits via POST /api/submissions with a file upload (PDF/Word/etc.)
  • After submission, student can view their submitted file and any feedback/grade from the doctor
  • Overdue assignments are flagged with a red indicator
  • Submission downloads: GET /api/submissions/download/:id
📖

Course Materials

  • GET /api/materials/my-materials — returns all materials for enrolled courses
  • Materials include: lecture notes, PDFs, slides, references — uploaded by doctors/assistants
  • Filtered by course; student only sees materials for their enrolled courses
  • Download any material via GET /api/materials/:id
  • Material cards show: title, course, upload date, file type icon
👤

Student Profile

  • View and edit: first name, last name, email, major, department
  • Upload profile picture: POST /api/students/profile/picture (multipart upload)
  • Change password with current-password verification
  • View academic info: student ID, year, semester, group, section (read-only — set by admin)

  • Refresh button to reload if the master schedule was regenerated
  • Doctor sees all 3 groups' lectures (e.g. Groups 1–3 or 4–6 per course)

Schedule Generator Algorithm

The core algorithm in scheduleGenerator.js places all lectures and sections without any conflicts. It uses a randomised greedy placement approach.

📐

Key constraints enforced simultaneously: No two events in the same venue at the same time · No staff member teaching two slots simultaneously · No student group having two slots at the same time · Doctor max 3 days/week · Assistant max 3 days/week, max 5 sections/day

  • Step 1 — Build Pools: Reads venue list from config (or defaults). Separates amphitheatres (lectures), labs and rooms (sections)
  • Step 2 — Pass 1: Lectures. For each course × each of 6 groups: shuffle WORKING_DAYS and TIME_SLOTS randomly → iterate until a free (day, time, amphitheatre) combination is found that satisfies all constraints → place and mark as used in the Registry
  • Step 3 — Pass 2: Sections. For each course × each of 12 group×section jobs (shuffled): round-robin through assistants → shuffle days and times → find a free lab/room slot → place it
  • Conflict Registry: A hash-set with three key types: V|venue|day|start (venue conflict), S|staffId|day|start (staff double-booking), G|group|section|day|start (student conflict)
  • Randomisation: Fisher-Yates shuffle on days and slots before every placement → slots are spread evenly across all 6 days instead of clustering at the start of the week
  • Result: 108 slots for 6 courses (6 lectures/course + 12 sections/course) with 0 conflicts
// Simplified generator pseudocode
for each course:
  for group = 1..6:
    days = shuffle(WORKING_DAYS)
    slots = shuffle(TIME_SLOTS)
    for day in days, slot in slots:
      if registry.canPlace(venue, doctor, group, day, slot):
        registry.place(...) → slots.push({...}) → break

8. Super Admin Exclusive Features

Super Admin Only
🔧

Schedule Configuration Editor

  • Edit all scheduling parameters: working days, time bounds, session durations, venue list
  • Saved to the ScheduleConfig singleton document in MongoDB
  • Version counter increments on every save so stale cached schedules can be detected
  • Changes take effect on the next schedule generation
⚡

Generate Master Schedule

  • POST /api/schedule/generate-master — runs the full algorithm for all active Year 2 Spring courses
  • Can filter by year and semester to generate only for a specific cohort
  • Result is upserted into the MasterSchedule collection
  • Response includes slot count, warning count, and a list of any unscheduled items
  • Each generation is randomised — re-generating produces a different valid timetable every time
🗑️

Delete Master Schedule

  • Button visible only to superadmin on the Timetable tab, only when a schedule exists
  • Confirmation dialog with warning before deletion
  • DELETE /api/schedule/master — removes the MasterSchedule document AND clears all per-student Schedule documents
  • All staff and student schedule views immediately show "No schedule yet" after deletion
🚀

Force Push Schedules

  • Force Group Schedule: Push schedule to all students in a specific Group + Section combination
  • Force All Staff Schedules: Save each staff member's personal slots (filtered from master by their ID) as their Schedule document — they can then view it without generating manually
  • Generate All Students: POST /api/schedule/generate-all — bulk generate and save schedules for all students based on their group/section assignments
✅

Recommended workflow: Generate Master → Push All Staff Schedules → Force Group schedules for each group → All users can now see their personal timetable

📋

Course Staff Assignment

  • For each course, superadmin assigns which doctors (max 2 per course) and which assistants (max 2 per course) will teach it
  • Stored as scheduleDoctors[] and scheduleAssistants[] on the Course document
  • The generator reads these arrays when building lectures (doctors) and sections (assistants)
  • Admin can remove a course from schedule generation by clearing its staff assignment
  • Only courses with at least one doctor OR assistant are included in generation

Security Architecture

  • JWT Auth: All routes (except login/register) require a valid Bearer token. Tokens expire in 7 days
  • Password Hashing: bcrypt with 10 salt rounds — passwords never stored in plain text
  • Role Guards: isAdmin, requireSuperAdmin middleware on all sensitive routes
  • Auto-retry: Frontend automatically retries failed requests once on 5xx errors
  • Cache Busting: After any mutation (POST/PUT/DELETE), related cache keys are cleared so stale data is never shown
  • Keep-alive: Frontend pings the backend every 55 seconds to prevent cold starts on free hosting tiers
⚡

Frontend Performance (Caching & Deduplication)

  • 60-second GET cache: Read-heavy endpoints (courses, grades, schedule) are cached in memory for 60 seconds so switching tabs is instant
  • In-flight deduplication: If two components request the same endpoint simultaneously, only one HTTP request is made; both get the same promise result
  • Smart cache invalidation: Each mutation calls clearCache('key') with a prefix to bust only related cache entries (e.g. grading a student only clears grade cache, not course cache)
